# Reference routing

Reference 的作用是减少猜测，不是增加上下文数量。先判断当前改写缺少哪类证据，再选择材料。

## Reference 类型

| 类型 | 可以影响 | 不能单独证明 |
|---|---|---|
| 当前用户要求 | 本次目标、边界、语气和交付方式 | 原文没有提供的事实 |
| 目标原文 | 事实、观点、归属、时序、判断强度 | 作者在其他场景中的稳定文风 |
| 作者认可的前后对照 | 信息取舍、叙述顺序、判断分寸、停笔位置 | 新文章中的事实 |
| 作者正样本 | 声音、节奏、读者距离和常用结构 | 作者对当前主题的观点 |
| 作者风格画像 | 跨样本的稳定倾向 | 每种体裁都必须遵守的硬规则 |
| 原始 session 语料 | 口语习惯和协作方式的弱证据 | 公开写作风格；其中的粘贴内容也未必由作者创作 |
| 事实资料 | 名称、数字、机制、时间和限制条件 | 文章声音和作者立场 |
| 平台或体裁样本 | 篇幅、格式、读者预期和信息密度 | 作者声音和事实判断 |

## 选择顺序

1. 先服从当前用户的明确要求。
2. 用目标原文锁定事实、归属和核心观点。
3. 有作者认可的前后对照时，优先用它理解作者做了哪些写作决策。
4. 没有对照时，选择与当前体裁、主题距离最近的作者正样本。
5. 风格画像只用于补充稳定倾向，不覆盖更具体、更新的样本。
6. 事实不确定时再查事实资料，不要因为要“合理使用 reference”而无目的搜索。
7. 只有目标平台确实影响交付时才看平台样本。

同类 reference 冲突时，优先级通常是：当前明确修改 > 作者最新认可版本 > 同体裁样本 > 同主题样本 > 综合风格画像 > 原始 session 统计 > 通用写作规范。

## 去哪里找 reference

按这个顺序发现材料：

1. 用户在当前请求中附带、粘贴或明确指向的文件；
2. 当前对话中被用户亲自修改、接受或拒绝过的版本；
3. skill 内已经整理好的作者 voice reference；
4. 用户明确给出的 reference 目录中，与当前体裁和主题最接近的少量样本；
5. 项目中明确命名的写作规范、style guide 或同系列文章；
6. 为核实不稳定事实而查到的权威来源。

不要仅因为同目录存在很多文章就全部读取。先看文件名、元数据和体裁，选择最可能影响当前决策的两三份；读取后发现用途不明确就停止使用。

不要为了模仿作者去搜索互联网。公开网页只有在用户明确把它作为文风样本，或改写需要核实事实时才有价值。事实查证优先使用一手来源，其语气不得进入正文。

不要直接扫描全量 session 来完成普通改写。已有风格画像或作者 reference 时优先使用整理后的结论；只有用户明确要求重新学习文风，才回到原始语料，并区分用户原话、粘贴内容和 AI 生成文本。

## 为每个 reference 建立用途卡片

不必输出给用户，但改写前要在心里写清楚：

- `role`：事实、作者声音、前后对照、平台还是读者背景；
- `reliable_for`：它能可靠影响哪些决定；
- `do_not_infer`：不能从中推断什么；
- `signals`：最多记录三到五个真正有用的信号；
- `conflicts`：它是否和目标原文或用户当前要求冲突。

如果无法说明某个 reference 会影响哪个具体决定，就不要加载或引用它。

## 如何阅读前后对照

不要只看改后版本增加了哪些词。比较：

- 开头从抽象背景换成了哪个具体事实；
- 哪些信息被删除，说明作者认为读者不需要什么；
- 哪段被前移，说明作者认为什么最值得先知道；
- 哪些确定判断被收窄，哪些模糊判断被说得更直接；
- 作者在哪里停止，没有继续补充哪些“完整但没必要”的内容；
- 表面修改背后是否存在视角、所有权或读者关系的变化。

把这些差异转写成当前文章的决策，不要复制原句或把每篇文章套成相同结构。

## 防止 reference 污染

- 不把事实资料的正式语气带进作者正文。
- 不把作者样本中的项目、经历、结论带入无关文章。
- 不因为作者正样本常用第一人称，就给中性原文补写“我做过”“我踩过坑”或“我得到这个结果”。
- 不把用户粘贴的 AI 回复或第三方文章当作用户文风。
- 不因为平台样本流行就添加标题党、悬念或金句。
- 不从单个错字、随机空格或一次大小写推断稳定规则。
- 不把统计频率直接当成写作要求；高频短指令不等于长文也应该全是碎句。
- 不用模型常识悄悄补全教程步骤。新增命令、参数、机制和安全建议属于内容扩写，需要用户授权或事实来源。
- “原文提到加入 wheel 组”不能自动扩写成一条新的命令；“修改 bashrc 后需要生效”也不能自动补入 `source ~/.bashrc`。推导合理不等于原文已经包含。

## Reference 不足时

Reference 只能支持表面特征时，保护原文并缩小改动，不要假装已经掌握作者声音。事实 reference 缺失但改写不改变事实时，可以继续；改写需要新增或强化事实时，先查证或询问。
