# Rewrite

## 目标

让现有中文技术写作更值得目标读者读下去，同时仍然像原作者写的。

不要把“更好”默认理解成更正式、更完整或更像一篇标准文章。改写可以通过删除、调序、收窄判断、恢复作者视角或降低阅读成本完成。只有原文的视角、因果顺序、信息取舍或结构妨碍表达时，才进行结构性重写。

## 把 reference 当作不同类型的证据

先阅读全文，再选择 reference。不要因为文件存在就全部加载，也不要把所有 reference 都当作文风样本。

当用户提供、指向或要求使用 reference，或者要求贴近某位作者的声音时，完整阅读 [references/reference-routing.md](references/reference-routing.md)，按其中的类型和优先级选择材料。

当文章要以当前用户本人的声音写作，并且没有更具体、更新的作者正样本时，完整阅读 [references/chengyongru-voice.md](references/chengyongru-voice.md)。它只约束声音和写作决策，不能替代目标文章中的事实。

始终遵守这些边界：

- 当前用户要求高于所有 reference。
- 目标原文决定事实、归属、时序和核心观点。
- 事实资料只能支持事实，不能自动决定声音。
- 作者样本只能支持声音，不能把样本里的事实带进目标文章。
- 作者声音不能授权虚构第一人称经历。原文没有表明作者亲自做过、踩过坑或得到过某个结果时，不要补写“我最近”“我一般”“我之前遇到”“我当时得到”。
- 平台样本只能帮助判断篇幅、结构和读者预期，不能覆盖作者观点。
- 用户粘贴或引用的文字不一定由用户创作，未经确认不能当作文风证据。

最有价值的 reference 通常不是“像这种感觉”的泛泛样本，而是作者亲自修改并认可的前后对照。学习作者改了什么、删了什么、为什么停在这里，不要只复制改后版本的词语和标点。

## 先判断写作关系

在心里确认：

- 谁在写，写给谁看；
- 文章要记录、解释、说服、复盘，还是留下判断；
- 读者已经知道什么，不必再解释；
- 读者读完后应当多获得什么；
- 文中的观察、行动、想法和结论分别属于谁；
- 作者站在哪个时间点叙述，事件在这个时间点是刚发生、仍在进行，还是已经结束。

不要机械地把这些答案写进开头。身份和上下文已经清楚时，不要重新自我介绍。事实归属无法确认且会改变含义时先查证，仍不确定再询问用户。

区分事件发生、原文写作和本次改写的时间。文章从即时记录变成事后复盘后，不要继续用“最近”“刚刚”“前几天”制造仍在现场的错觉；只有作者确实站在那个当下写作时才保留。优先删除失效的相对时间，或用事件、版本和状态变化定位。不要机械换成“以前”“在……之前”“我最开始……”，换一种时间说法仍可能只是多余的开场。

## 先保护原文

默认原文已经有值得保留的部分。改写前找出并保护：

- 必须保持准确的事实、数字、名称、代码、链接和因果判断；
- 作者已经写得自然的句子、转折、例子和结论；
- 原文的视角、判断强度和不确定性；
- 稳定的大小写、中英文混排、标点密度和口语习惯。

把原文中的代码块、命令、参数、路径、版本号和预期输出视为 locked content。默认可以移动位置、修复 Markdown 围栏或调整周围说明，但不能新增、删除或改写其技术内容。只有用户明确要求修复或补全技术内容，并且变更得到可靠证据支持时才能修改。

不要先把原文规范化，再尝试补回作者声音。API、函数、配置键、命令和路径必须精确；普通技术名词跟随作者习惯。作者写 `windows`、`agent`、`docker` 时，不要仅为品牌规范改成大写。

## 选择正确的改写深度

### 局部改写

当原文的观点、顺序和叙述关系已经成立时，优先删除重复，修正不自然句子，压缩弱段落，保留整体结构。不要为了体现工作量重写。

默认保留体裁。教程仍然是教程，调查记录仍然是调查记录，短帖仍然是短帖。作者 voice 可以改变信息取舍、节奏和判断分寸，但不能凭空把中性教程变成个人经历，或者把随手记录变成完整故事。

### 结构性改写

只有出现这些情况时才重排全文：

- 报告式结构遮住了作者真正想表达的判断；
- 重要因果关系出现得太晚，读者无法理解为什么值得关心；
- 文章列出了过程，却没有写出哪些证据改变了判断；
- 目标平台、读者或文章任务已经改变；
- 作者明确要求采用某个 reference 中的叙述方式。

结构性改写不是增加标题和段落，而是重新确定信息出现的顺序。工程调查常见的自然推进是：真实问题或异常现象 → 第一反应 → 副作用或反例 → 对问题层级的新判断 → 验证证据 → 有边界的结论。只有材料本身具备这条变化时才使用，不要套模板。

## 在写作决策层模仿，而不是表面模仿

使用作者 reference 时，依次观察：

1. 作者选择从哪个事实开始；
2. 哪些背景被省略，默认读者已经知道；
3. 作者何时提出判断，何时用反例修正；
4. 判断说到多满，哪里保留不确定性；
5. 段落如何推进，什么时候停笔；
6. 最后才观察词语、大小写、空格和标点。

不要机械加入“其实”“当然”“我觉得”来制造口语感。不要故意添加错字、重复空格或不完整句。真正需要保留的是作者的思考顺序、信息密度、读者距离和判断分寸。

## 删除也是改写

一句话即使真实、通顺，也可能只是重复前文、解释读者已经知道的背景、记录与结论无关的过程，或用较长篇幅重复已经成立的判断。优先删除这些内容，不必为了维持篇幅再补一段。

不要删掉：

- 支撑核心判断的证据；
- 必要的推理变化和反例；
- 作者独有的观察；
- 读者理解后文所需的上下文；
- 能改变结论强度的限制条件。

## 写出自然的中文推进

每个段落都应提供新的事实、解释、证据、判断或后果。真正出现矛盾、边界或新证据时再转，不要用连接词制造虚假的起承转合。

优先使用能说明具体对象、原因和后果的主语与动词。相邻句反复使用“不是 A，而是 B”“一方面/另一方面”或相同句式时，调整论证关系，不要只换同义词。

开头能直接说出现象、机制或后果时，删除“最近发现一个问题”“这里有个不自然的地方”“事情要从……说起”这类只预告正文的元开场。不要再换一句“在……之前有个问题”继续铺垫。让具体事实先出现；例子已经能让读者感到奇怪、麻烦或出戏时，不要提前替读者下判断。

不要把个人发现改成新闻稿、产品公告或行业综述。不要凭空添加悬念、宏大判断、总结金句或圆满结局。观点讲清楚之后就停。

## 尊重技术读者

保留必要的技术密度，不为流畅删掉关键名称、数字、命令、日志和证据，也不补充读者能够自行推断的基础背景。

不要把结论写得比证据更强。reference 与原文冲突时，不要偷偷调和；优先保持原文事实和作者当前明确判断，必要时说明冲突或请求确认。

Rewrite 默认不是事实扩写。除非用户明确要求补充、查证或完善内容，不要新增原文没有的命令、步骤、参数、预期输出、故障原因、安全警告或个人经历。确实存在影响正确性的缺口时，先查证，再明确区分“原文已有事实”和“本次补充”。

“按照常识应该执行”“为了让教程完整”“从现有步骤可以推导”都不能自动授权新增内容。一条逻辑上合理但原文没有的 `source`、`systemctl`、安装或清理命令，仍然是事实增量。

## 执行顺序

1. 阅读目标全文和用户要求，确认文章任务、事实归属与叙述时点。
2. 列出必须保留的事实、原文已有的优点和不可改变的声音特征；单独锁定所有代码、命令、路径、数字、版本、输出和第一人称经历。
3. 发现可用 reference，按 routing 规则只读取真正相关的部分。
4. 为每个 reference 明确：它能影响什么，不能证明什么。
5. 判断使用局部改写还是结构性改写，并明确最影响阅读价值的一两个问题。
6. 依次考虑保留、删除、移动、改写，只有存在真实信息缺口时才补充。
7. 先处理信息取舍、因果顺序、视角和判断强度，最后才调整句式与表面风格。
8. 将新版与原文逐段比较；没有明确净收益的实质改动要恢复。
9. 做一次 reference 审计，确认没有抄入无关事实、口号、措辞或其他作者的声音。
10. 做一次事实增量审计，逐项比较 locked content。任何新出现或被改写的命令、代码、路径、数字、版本、输出、机制、经历和因果关系，没有用户授权与事实来源就删除或恢复原文。
11. 连续通读全文，检查它是否自然、准确，而且仍然属于原作者。

## 最终判断

- 新版是否让读者更容易获得原文真正有价值的事实和判断？
- 原文已有的直接、克制、准确和个人声音是否仍然存在？
- reference 是否影响了正确的层级，而没有越权改变事实或观点？
- 新版是否虚构了第一人称经历，或用常识补入了未经要求的新步骤？
- 原文中的 locked content 是否保持完整，新版是否出现了原文没有的命令、代码行、路径、数字或输出？
- 新版是否学习了作者的选择方式，而不只是复制口头禅和标点？
- 删除是否降低阅读成本，同时保留了证据、推理变化和限制条件？
- 事实归属、时序、不确定性和技术含义是否准确？
- “最近”“刚刚”“此前”等时间词是否符合文章最终的叙述位置？
- 开头是否只在预告“有个问题”，而没有提供具体事实？
- 大小写、中英文混排和口语习惯是否得到尊重？
- 新版是否确实优于原文，而不只是更完整、更正式或更“像文章”？

如果不能明确说明新版整体更好，就保留原文或缩小修改范围。
