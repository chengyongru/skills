# GitHub Review Submission Reference

Load this only after publication is authorized, at least one useful finding exists, and line-specific findings have been anchored with `references/line-comments.md`.

## Rules

- Submit `COMMENT` reviews only.
- Never submit `APPROVE`.
- Do not submit `REQUEST_CHANGES` unless the user explicitly asks for that exact event.
- Do not create a GitHub review when there are no blockers/useful comments.
- Follow repository norms or the PR discussion language for public feedback.
- Disclose automation. Default to `> Automated review generated by <REVIEW_TOOL>.`, replacing `<REVIEW_TOOL>` with the actual tool or agent name, unless repository policy specifies another wording.
- Prefer inline comments for line-specific findings. Use the review body for cross-cutting findings that cannot be anchored honestly to one changed line.
- Keep risks/questions distinct from confirmed defects.

## Build and submit payload

Write the payload builder in the operating system's temp directory first. Do not use `python -c`; Markdown/backticks/quotes in review bodies are easy to break in shell strings.

```python
# <temp-dir>/build_review.py
import json
import tempfile
from pathlib import Path

body = (
    "> Automated review generated by <REVIEW_TOOL>."
    "\n\n"
    "Review body here. Keep it concise."
)

comments = [
    {
        "path": "<file path>",
        "line": <actual file line number>,
        "side": "RIGHT",
        "body": "Inline comment here",
    },
]

payload = {
    "body": body,
    "comments": comments,
    "event": "COMMENT",
}

payload_path = Path(tempfile.gettempdir()) / "pr_review_payload.json"
with payload_path.open("w", encoding="utf-8") as f:
    json.dump(payload, f, ensure_ascii=False, indent=2)

print(payload_path)
print(f"Payload written: {len(comments)} inline comments")
```

```bash
python3 <temp-dir>/build_review.py
gh api repos/<OWNER>/<REPO>/pulls/<N>/reviews --input <printed-payload-path> --jq '{id, state, html_url}'
```

Save the returned URL for the final local report.

## Fallback: diff position

Prefer `line` + `side`. If GitHub rejects the anchor even though the line is in the changed file, fall back to legacy `position`.

```bash
gh pr diff <N> --repo <OWNER>/<REPO> > <temp-dir>/pr<N>.diff
```

`position` is the 1-based index of changed lines within that file's diff section, counting changed/context diff lines as GitHub expects and excluding file headers. Use this only as a fallback; actual file line + side is simpler.

## Common mistakes

| Mistake | Fix |
|---------|-----|
| Publishing from a read-only review request | Post only when the user or repository policy authorized publication |
| Using `APPROVE` | Never approve from this skill |
| Posting a clean-review comment | Do not post when there are no blockers/useful comments |
| Missing disclosure | Use a truthful automated-review disclosure |
| Forcing a cross-cutting concern onto an unrelated line | Keep it in the review body |
| Using `python -c` for payload | Write a temp script/file first |
